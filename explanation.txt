The main objective in this project is to minimize the VRAM usage. GPUs are commonly memory bound in GNN scenarios, it's even more common in scenarios in which we are scaling the distance in which a node is considered neighbor, the reason being that the edge tensor grows cubically.

# =======================================================
# Section 1. Main node identification and feature load
# =======================================================

This section involves the strongest decision in the whole project, which is the main focus of the kernel. Initially, it was proposed an edge-focused kernel, meaning that what the kernel does is focus on edges and calculate two nodes at the same time. This path is best when talking about small graphs but it doesn't scale at all, the reason being that by focusing on edges, the kernel has to do an atomic add for the partial messages and coordinate update of two nodes. The more edges a node has the more probable is that one thread has to wait for another to finish.

On the other hand, the strategy chosen in this project is a node-focused kernel. The reasoning behind this decision is the following: in small scenarios an edge-focused kernel would win but there wouldn't be much of a difference. In a small scenario the speedup difference might be from 3 to 4, but in a big scenario, the speedup would be of tens of times bigger than the other focus.

That being said, the main focus is the node, but specifically the kernel works around one node. The focus is a scenario in which a node has a lot of edges, meaning that one node has enough working with multiple nodes at a time would probably result in a register spilling or bad cache usage.

Another important design decision is the way the node works with its neighbors. If we created tensors with all the features of all the neighbors the memory would still be endangered without a necessary reason. By adding a for loop that iterates between neoghbor blocks, the max memory is limited to the GPU's best configuration (thanks to the autotune).

Here the code is fetching the main node properties, which are the node features and its coordinates. The way of fetching data in Triton generally consists on the following steps:
    
1. sequence. In this environments the focus is shifted from a single value calculation to matricial calculations. For this to work, the variables must be matrices. This means that when fetching data, the requested data should also be in a matricial shape. The dimensions of the matrices will be commonly be defined by sequences [0, 1, 2...].
These sequences will also commonly be generated by the function arange(start, finish), which is a function that will replace the value in compilation time with an array of the given dimensions.

2. address calculation. The pointers that the kernel receives are commonly shared with all the other instances of       kernels because the pointer logic is usually transparent to the user. Consequently each kernel has to calculate     their respective offset. In this case, this calculation is rather simple because we are only fetching the info of one node but in the future this calculations will be more complex.

3. mask. GPUs commonly work with fixed dimensions of the power of 2. This means that most of the time, the tensors     will have a padding which will contain garbage. One clear example that in this code always applies is the coordinate tensor.
When doing the sequence of coordinate addresses, the arange function only accepts a dimension by the power of 2. This means that the coordinate tensor will be rounded from 3 columns to 4. This fourth column is empty or contains garbage that should not be processed, thus the mask "coord_sequence < 3".

4. load. Now this function meshes the previous calculations and loads the requested data. One important concept to point out is the "other" parameter. When doing the mask what the code actually does is just a mask, so the load requests info about how to deal with this marked data. In this case what the "other" parameter does is indicate the value that will substitute the marked values. This is an important distinction, the code IS processing the data, so one of the objectives should be reducing the padding as much as possible. It is also important to clarify that when storing the data, this mask should also be used or the code will be storing wrongful data.
    
Now going back to the code snippet, the objective is to obtain the tensors of the features and the coordinates of the main node. These two tensors are exclusively for the main node, meaning that the number of rows should be 1. For the features tensor, the number of columns should be F_NODE and for the coordinates tensor it should be 4 as explained before. Note that the F_NODE might not actually be the actual number of features, thus the masking.

The properties of this node will be used for processing all the neighbors, meaning that the code must somehow be able to consult these properties for all the nodes. This necessity leads to a commonly used technique when dealing with tensors of different dimensions called broadcasting.
Broadcasting allows the code to multiply two matrices that apparently aren't compatible.

[GENERATE EXAMPLE OF BROADCASTING]

In this example, the y dimension of the main node features is lengthened in order to fit the dimensions of the tensor of properties of the main node neighbors.
This operation is usualy done implicitly, but in this code the broadcasting is being done on purpose, this offers some benefits and drawbacks.
The main drawback of explicit broadcasting is the memory usage. In this case VRAM is left untouched and what the code is using is registers. As stated before, optimizing VRAM means using registers, which have the risk of overflowing and end up using VRAM. What this means is that registers are a precious resource that should be used carefully because a bad usage may tank performance. By broadcasting explicitly, the code is using some of the valuous registers, which on paper sounds like a major drawback, but it is worth it.
It is true that the main objective is reducing memory usage, but take all the decisions just focusing on memory consumption isn't worth it in some scenarios. There already are implemented measures that deal with handling register usage, which means that these decisions are kind of a balance. By cutting off in some corners, the code can afford using registers when the benefits are worth it.
In this case, broadcasting allows the usage of a GPU component called Tensor Cores. Tensor Cores are a dedicated hardware component that allows calculating matrix multiplications in a single clock cycle. The drawback of this component is that the data shape mustbe very specific, so broadcasting isn't allowed in this case, thus the reason why it is done before.

# Section 1.5: Weight Fetching (Load ONCE into Registers)
# =======================================================

The next section will be a for loop that will iterate neighbor blocks. Some if these iterations will share variables. If the code weren't to load these univariant variables, there would be extra calculations and more importantly, extra memory loads (which would be in a cache but nonetheless useless when it's already in a register).
The main shared variables are the weights values of the 3 MLPs. Note that technically the main node properties are also univariant variables, so this section is a natural extension of the previous one. Each MLP has its own dimensions so they will be explained separately.

MESSAGE MLP
The message MLP has as an input the features of the two nodes, the distance as a single scalar value and the edge features. 
Computing this MLP exactly as stated would create a tensor of huge dimensions, while also having to align all the data. One common optimization is calculating the partial output, computing each multiplication separately. Another possible optimization possible is also computing the main node weigths BEFORE the loop, which would reduce the number of calculation of these exact weights from E to 1. This last optimization isn't done because it has some major drawbacks.
First of all, this would overcomplicate the code, the operations would be scattered along the code because we would be able to optimize for each MLP this step. This extra confusion isn't generally worth the complexity. Moreover, the calculation uses Tensor Cores whereas the optimized version doesn't, so the computation gap isn't noticable. 
Finally, the compiler automatically manages data, trying to minimize register spilling. By making the computation of the weights before the loop, the code is creating a tensor of HIDDEN_LAYERS dimensions (which tend to be bigger than the original dimension) and by doing it at the beggining, the code is obligated to reserve across the whole execution these registers. This by far is the biggest drawback.
Returning to the weights computations, before dealing with memory fetching, one common extra step that helps is trying to visualize the figure of the tensors. In this case, the length of the input corresponds to the height of the weights tensor, which is F_NODE + F_NODE + 1 + F_EDGE, where 1 is the dimension of the distance. The width will correspond to the dimension of the hidden layer.
As stated before, this weights will be partitioned and calculated separately. The code should reflect this by adding offsets to the address. For the weight of the first node there is no offset, for the second node there is the width of the previous tensor (F_NODE x HIDDEN_LAYER). For the distance we add the same offset. Finally for the edge features we add all the previous offsets plus 1 x HIDDEN_LAYER (which corresponds to the distance).
Note that as stated before, in the registers, the tensor isn't aligned (which is what the code aims for) but by only providing one weight pointer to the kernel, this tensor must be aligned in disk, which is good because it provides memory locality without drawbacks.
The code repeats the process for the movement MLP. The input of this MLP is the output of the message MLP, so the height of the tensor is OUT_FEATURES and the length is the movement's hidden layer dimension. After this first layer, the process is again repeated, this time for the weights of the hidden layer. This additional tensor has dimensions [HIDDEN_LAYER x OUTPUT_LAYER]

Now it's turn of the univariant backward variables. In this case, the code not only loads variables, but it also reserves memory for posterior storing.
Training an EGNN layer requires all the variables from end-to-end, which means that we need the input also. This input is the output of the previous layer, which means these are correctable values, which means that we require a gradient to chain the loss. This input and its gradients aren't computed by the current layer, so the code loads them.
All the rest of the values are computed by this layer, so instead of loading them, the code initializes them.
The input of the layer is: the two node features, the distance and the edge features. The node features are the ouput of the node MLPs from the main node MLP and the neighbor node MLP. 
When doing the training, Pytorch ensures that the kernels are executed in a determined order, which means that all the node process will be syncronized refering the layer. This is crucial because the gradient of another layer is an addition of all the corrections of all the neighbors. So the kernel confidently loads the gradients of the neighbors and main node, relying on this syncronization.
